# Codex Instructions For `iam-sdk-js`

These instructions are for AI agents and future maintainers working on this repository. Follow them before changing code.

## Project Context

- This is an npm workspaces monorepo for TOTVS Cloud IAM frontend SDK packages.
- Public packages:
  - `packages/core`: `@totvs-cloud/iam-sdk`, framework-agnostic core SDK.
  - `packages/react`: `@totvs-cloud/iam-sdk-react`, React adapter built on top of the core SDK.
- `examples/core-manual-test` is an isolated Vite/React manual test app published to GitHub Pages. It is not a public npm package and must stay outside root workspaces.
- The SDK mirrors the main behavior of `iam-sdk-python` while adapting authorization to the frontend BFF.
- The frontend authorization RFC is `RFCs/text/0009-autorizacao-frontend-iam-cedar.md`.
- The real BFF implementation lives under `iam/src/infra/aws/rust/frontend_authorizations/`.
- Browser-facing APIs must never call the Cedar Agent directly.

## Non-Negotiable Authorization Contracts

- Frontend authorization must call `POST {endpointAuthzBatchEvaluate}`.
- The endpoint should normally point to the full batch validation URL `/frontend/authorizations/evaluate`; the SDK must not append path segments.
- The request body is exactly `{ checks: [...] }`; do not send `tenant` in the body.
- The BFF extracts tenant from `ext.tenant` in the JWT.
- Use `Authorization: Bearer <token>` for BFF, AuthN and Control Plane calls.
- Authorization checks v1 use direct actions in `service:action` format, for example `iam:listUsers`.
- Capability keys (`key`) are not supported in v1. Keep throwing `UnsupportedCapabilityKeyError` unless the BFF contract changes.
- The BFF limit is 50 checks per batch. Keep chunking batches above 50.
- Fallback authorization endpoints are used only for transport failures or timeouts. HTTP responses from an endpoint are authoritative.
- If multiple checks would produce the same response key, the core SDK must generate aliases before sending the request.

## Package Rules

- `@totvs-cloud/iam-sdk` must remain framework-independent and must not import React or other UI frameworks.
- `@totvs-cloud/iam-sdk-react` must use `@totvs-cloud/iam-sdk` as a peer dependency and should not duplicate core AuthN/AuthZ logic.
- React adapter APIs must stay thin: `AuthzProvider`, `useIamClient`, `useCan` and `Can`.
- React adapter must not implement `services`, `loadServicePermissions` or capability-key flows until the real BFF supports them.
- Keep runtime dependencies minimal. Prefer platform APIs and injectable `fetcher`.
- Avoid Node-only runtime APIs in source files that run in browsers.

## Code Organization

- Core SDK:
  - `packages/core/src/client`: public `IamClient` facade and factory.
  - `packages/core/src/authn`: login, assume role, token validation and current roles.
  - `packages/core/src/authz`: BFF authorization, batch chunking, aliases and cache integration.
  - `packages/core/src/cache`: in-memory authorization cache.
  - `packages/core/src/http`: fetch wrapper, timeout, JSON parsing, typed HTTP/transport errors and fallback behavior.
  - `packages/core/src/iam`: IAM Control Plane helper methods matching the Python SDK.
  - `packages/core/src/errors` and `packages/core/src/types`: public errors and types.
- React adapter:
  - Keep hooks/components small and based on the core public API.
  - `useCan()` should check the core cache first, dedupe simultaneous checks through the provider, and deny by default on errors.
  - `<Can>` should render children only when allowed.

## Testing Requirements

Run these before finishing code changes:

```bash
npm run lint
npm run typecheck
npm run test
npm run build
```

If the manual app is touched, also run:

```bash
npm --prefix examples/core-manual-test run typecheck
npm run build:manual
```

Add or update tests when changing:

- BFF request body or headers.
- Authorization alias generation.
- Batch chunking.
- Cache keying, hydration, TTL or invalidation.
- Fallback endpoint behavior.
- AuthN response handling.
- Control Plane paths, payloads or URL encoding.
- React provider/hook/component state behavior.
- Public exported types, package exports or errors.

Tests should use injectable fetchers or local mocks; do not hit real IAM services.

## Versioning And Release

- Follow SemVer per package.
- New tags are package-scoped:
  - `iam-sdk@X.Y.Z` for `@totvs-cloud/iam-sdk`.
  - `iam-sdk-react@X.Y.Z` for `@totvs-cloud/iam-sdk-react`.
- Historical `vX.Y.Z` tags belong to the original single-package core releases.
- Update the package `CHANGELOG.md` for user-visible changes.
- Use `.github/workflows/versioning.yml` to create tags and GitHub Releases.
- `.github/workflows/release.yml` publishes only the workspace identified by the tag.
- Keep `dist/` generated by package build scripts; it is ignored by Git and should not be manually edited.
- Do not put registry tokens, IAM credentials, service tokens or JWTs into the repo.
- The release workflow expects `NPM_TOKEN` in repository secrets.

## Security Notes

- The SDK is a UX authorization helper, not an enforcement layer. Backend services remain the authority.
- Do not expose Cedar Agent service credentials to frontend code.
- Do not log tokens, secrets or full authorization headers.
- Authorization cache decisions are per token/principal context; be careful with cache invalidation changes.

## Dependency Policy

- Prefer standard APIs and small code over adding dependencies.
- If adding a dependency, explain why it is needed and add tests around the behavior it supports.
- After dependency changes, run `npm audit --audit-level=moderate`.
- Avoid `npm audit fix --force` unless explicitly approved because it can introduce breaking toolchain upgrades.

## When Backend Contracts Change

If the BFF starts supporting capability keys or `evaluate-service`, update the SDK deliberately:

- Add tests for the new wire contract first.
- Keep direct `action` checks backward-compatible.
- Update READMEs and `CHANGELOG.md`.
- Revisit `AuthorizationCheck` types and `UnsupportedCapabilityKeyError`.
- Do not assume RFC text alone is the source of truth; inspect the implemented BFF code.
